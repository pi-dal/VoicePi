# External Processor Source Snapshot Design

**Status:** Proposed

## Goal

Allow the existing external-processor shortcut flow to capture the current text selection as a one-shot `Source` snapshot, pass that snapshot into the external processor as explicit reference context, and show a compact preview chunk in the floating panel without changing the current `stdin` contract.

## Background

VoicePi already supports one-shot external refinement through `Alma CLI`:

- the resolved refinement prompt is passed as command-line prompt text
- the live dictated text is written to `stdin`
- the floating panel shows a compact single-line recording or refining banner

That contract is stable and should remain stable in this phase.

The new requirement is not to make the selected text the main request. Instead:

- the user still speaks the live request
- the selected text becomes preset reference context
- the external agent must be able to tell that this reference context is a `Source`, not the active instruction

The `Source` must be captured at shortcut time as a snapshot. It must not track later selection changes.

## Product Decision

VoicePi should treat `Source` as a third input role alongside the existing two roles:

1. `prompt`
   - processing rules and behavior contract
2. `stdin`
   - the live dictated request for this invocation
3. `source`
   - preset reference context captured from the current selection when the shortcut starts

The key compatibility rule is:

- keep `stdin` meaning unchanged

This phase therefore does **not** move selection text into `stdin`, and does **not** require prompt authors to adopt a new template variable system.

Instead, VoicePi should extend the generated prompt with an explicit `Source` contract only when a non-empty selection snapshot exists.

## Core Semantics

### Snapshot Timing

When the external-processor shortcut starts a recording session, VoicePi should immediately inspect the current editable target and capture a `Source` snapshot if all of the following are true:

1. the focused target is editable
2. the target exposes a non-empty selected range
3. the selected text resolves to non-empty normalized text

If any of those checks fail, the session continues normally without a source.

### Snapshot Lifetime

The captured source belongs only to the current recording-processing session.

- it is immutable after capture
- it is discarded when processing completes or is cancelled
- it is not reused across later shortcut activations

### Source Meaning

The source snapshot is reference material only.

It is not:

- the current user request
- a second prompt
- a conversation history mechanism

The external tool should be told that:

- the live request arrives through `stdin`
- the `Source` is preset background context captured at shortcut time
- the tool may use the `Source` to ground the answer, but should not treat it as the current instruction to execute

## Prompt Contract

### Existing Contract

Today the external-processor path already relies on this semantic split:

- prompt explains how to process the input
- `stdin` carries the actual input text

That contract should remain the default.

### Extended Contract With Source

When a source snapshot exists, VoicePi should append a fixed contract block to the resolved refinement prompt. The block should be generated by VoicePi, not authored manually in each prompt preset.

Recommended shape:

```text
The live user request is provided via stdin.
The following Source is preset reference context captured at shortcut time.
Treat Source as background context, not as the user's current instruction.

<Source>
...
</Source>
```

This block should be appended after the resolved refinement prompt and before any target-language suffix already added by the external-processor prompt builder.

If no source snapshot exists, prompt generation remains exactly as it is today.

### Truncation Rule

The source sent to the external processor should have a bounded maximum length.

Recommended phase-1 default:

- maximum transmitted source length: `6000` characters after sanitization

If the source is truncated, VoicePi should append a short note inside the contract such as:

`Source was truncated to fit processor limits.`

This keeps the invocation bounded without changing the user-facing interaction model.

## Floating Panel UX

### Design Direction

The floating panel should remain compact. The source preview should appear as a small secondary chunk, not as a large content area.

Requirements:

- visually subordinate to the main transcript or refining status
- single-line preview only
- preview text should be derived from the start of the captured source
- long content should truncate with an ellipsis

### Preview Content

The preview chunk should display:

- a small `Source` label
- a preview string derived from the normalized source snapshot

Recommended preview rule:

- collapse internal whitespace to single spaces
- show the first `48` visible characters
- append `…` when truncated

Example:

`Source 这是用户刚才选中的一小段内容，后面继续截断显示…`

### Preview Lifetime

If a source snapshot exists, the same preview should remain visible through:

1. the recording state
2. the refining state

If there is no source snapshot, the panel should render exactly as it does today.

## Architecture

### Data Model

Add a lightweight per-session value type for captured source context.

Suggested fields:

```swift
struct CapturedSourceSnapshot: Equatable {
    let text: String
    let previewText: String
    let sourceApplicationBundleID: String?
    let targetIdentifier: String?
}
```

`previewText` is precomputed so the floating panel does not need to duplicate truncation logic.

This type does not need persistence in this phase.

### Capture Boundary

Capture should happen in `AppCoordinator` at the same point where the external-processor shortcut session is created, before recording begins.

That keeps the snapshot:

- tied to the same shortcut activation
- independent from later AX changes
- available to both UI and processor invocation logic

### Workflow Propagation

The active processing session should carry an optional `CapturedSourceSnapshot`.

That value should flow into:

1. floating-panel presentation
2. external-processor prompt generation

It should not flow into:

- standard injection-only recording flows that do not use the external processor
- result-review rewrite source tracking in this phase

## Integration Plan

### 1. Capture Selection Snapshot

Use the existing editable-target inspection path to attempt capture at external-processor shortcut start.

Normalization should reuse the existing text sanitizer so source text and refinement inputs follow the same cleaning rules.

### 2. Extend Floating Panel State

Add an optional source preview field to the floating panel content model.

If populated:

- slightly increase banner height
- render a secondary chunk row beneath or alongside the main transcript area
- keep the current compact width behavior

If absent:

- preserve current layout and measurements

### 3. Extend External Prompt Builder

Update external-processor prompt generation so it can optionally receive a source snapshot and append the generated `Source` contract.

This logic belongs with the external-processor prompt builder, not in the runner, because it is semantic prompt composition rather than process execution.

### 4. Preserve Runner Contract

Do not change:

- `ExternalProcessorInvocation`
- `ExternalProcessorRunner`
- `stdin` payload shape

The runner should continue to launch the process exactly as it does today, with the live dictated text written to `stdin`.

## Error Handling

### Missing Selection

If no valid selection exists at shortcut time:

- do not show an error
- do not show a source preview
- do not inject a source block into the prompt

This should behave as a normal external-processor invocation.

### Invalid or Empty Normalized Selection

If the selected text becomes empty after sanitization:

- treat it as no source

### Oversized Source

If the source exceeds the transmission bound:

- truncate before prompt composition
- keep preview generation independent from the full transmitted length
- annotate truncation in the generated contract

## Non-Goals

This phase does not introduce:

- persistent multi-turn conversation state for external processors
- hidden side channels beyond the existing prompt plus `stdin` contract
- prompt-template placeholders such as `{{source}}`
- UI for expanding or editing the source chunk
- reuse of source snapshots in result-review regeneration flows

## Testing Strategy

Use focused unit coverage.

### Prompt Composition

Add tests proving:

- source-free external prompt output remains unchanged
- source-aware prompt output appends the contract block
- target-language suffix still appears after the source block
- oversized source is truncated and annotated

### Snapshot Capture

Add tests proving:

- valid selection creates a source snapshot
- empty or whitespace-only selection does not
- snapshot text is sanitized before storage
- preview text is truncated deterministically

### Floating Panel Support

Add tests proving:

- preview text formatting collapses whitespace
- preview text truncates at the agreed limit
- no preview text is displayed when no source exists

### Workflow Safety

Add tests proving:

- external processor still receives the live dictated request through `stdin`
- the presence of a source does not change the live request payload
- non-external refinement paths remain unchanged

## Implementation Notes

The clean boundary for this feature is:

- `AppCoordinator` owns source snapshot capture and per-session lifetime
- floating-panel support owns compact preview formatting
- external prompt composition owns `Source` contract generation
- the runner remains transport-only

That keeps UI, semantic prompt assembly, and process execution separated and preserves the current external-processor integration contract.
